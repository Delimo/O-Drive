二、当前代码结构判断

1. 当前结构整体合理，适合继续加功能。

2. 当前结构的优点：
- 业务逻辑不再全部堆在一个文件里
- 页面渲染、事件处理、状态逻辑、服务层已经开始分层
- 后续继续补后台功能时，新增内容有明确落点
- 已经能逐步摆脱“改一处牵全局”的问题

3. 当前仍然存在的问题：
- `public/index.js` 已比上一轮更清晰，但仍有继续瘦身空间
- 个别 helper 仍留在入口文件中，职责边界还不够彻底
- 历史中文乱码文案还存在
- 设计预览模式已经补齐，但仍缺少更贴近真实场景的前端回归检查
- 预览模式下的写操作目前以禁用为主，后续若要演示更完整交互，可再评估是否补本地 mock 交互

4. 结论：
- 当前结构值得继续优化
- 现在继续拆，比重新堆回 `index.js` 更合适
- 未来新增功能应继续落在模块文件中，不应回退到入口堆积模式

三、规划总原则

1. 不回退到“单个超大入口文件”模式。

2. 新功能优先放入以下模块边界：
- API 请求：`public/js/api/`
- 服务层封装：`public/js/services/`
- 状态读取与派生逻辑：`public/js/state/selectors.js`
- 异步业务流程：`public/js/state/thunks.js`
- 页面事件：`public/js/events/index.js`
- 页面渲染：`public/js/render/`

3. `public/index.js` 只保留：
- 启动
- 注入
- 状态组装
- 根渲染
- 必要的全局 helper

4. 每次只推进一类工作，避免同时大改结构与大补功能。

5. 每完成一阶段，都要至少做一次：
- `node --check public/index.js`
- `npm.cmd run lint`
- `npm.cmd run build`

四、阶段规划

阶段 A：可部署版本稳定化

目标：
- 先保证当前版本适合部署给人看 UI
- 优先消除白屏、运行时断链、明显首屏异常
- 让首页、后台页、分享页都能稳定出基础界面

当前状态：
- 已完成大部分

本阶段已完成内容：
- 修复 `detectContentMode` 依赖断链
- 修复 `isProtectedEntry` 依赖断链
- 确认页面能渲染出首屏
- 确认构建通过

剩余建议：
- 再检查一轮是否还有同类型“拆分后漏注入”的 helper
- 清理极易影响部署观感的明显报错或空白区域

验收标准：
- 主要页面不白屏
- 没有阻断渲染的运行时错误
- `build` 与 `lint` 通过

阶段 B：本地预览体验增强

目标：
- 让没有后端联调时的本地预览也能看出真实 UI 水平
- 减少“全是报错态，所以看不出页面质量”的情况

当前状态：
- 已完成

本阶段已完成内容：
- 增加 `?mock=1` 前端设计预览开关
- 首页已补齐 mock 文件夹、文件列表示例数据
- 首页右侧文件详情可基于 mock 数据正常展示
- 后台页已补齐 mock 统计卡片与分享管理列表示例数据
- 分享页已补齐无需真实 token 的 mock 预览态
- mock 预览下的写操作已禁用，避免误打真实接口

建议内容：
- 增加 mock 数据模式，至少覆盖：
- 首页文件列表
- 右侧文件详情
- 后台统计卡片
- 后台分享管理卡片
- 分享页预览态

推荐实现方式：
- 方案 1：加一个简单前端 mock 开关
- 方案 2：本地预览时对关键 API 返回静态假数据
- 方案 3：保留当前正式逻辑，同时增加一个“设计预览模式”

优先级：
- 高

原因：
- 这是最快确认 UI 是否要重做的方式
- 比先补复杂功能更适合当前阶段

验收标准：
- 不接真实后端时，也能看到完整页面结构
- 首页、后台页、分享页都有可用示例数据
- 不影响正式部署逻辑

- 新增 `public/js/utils/text.js`

注意事项：

- 入口文件的职责更接近“编排层”
- 不引入新的依赖混乱

阶段 D：共享渲染与页面边界继续清晰化

目标：
- 让 `home/pages/modal/shared` 的边界更清晰
- 尽量避免页面渲染模块反向依赖入口函数

重点工作：
- 审查 `render/shared.js` 是否还应继续接管：
- 通用卡片片段
- 状态提示片段
- 小型统计块
- 详情键值块
- 空态片段

适合拆分的条件：
- 两个以上页面会复用
- 不含强页面语义
- 不需要直接读全局状态

验收标准：
- `pages.js` 更像页面拼装器
- `shared.js` 更像可复用视图片段库
- 共享渲染职责边界明确

目标：

建议优先顺序：
1. 上传体验增强
2. 搜索与筛选增强

- 更清晰的上传成功/失败反馈
- 更清晰的目录上传展示

搜索方向建议：
- 更可靠的搜索态提示
- 搜索结果空态优化
- 搜索与筛选、排序的配合表现优化

预览方向建议：
- Markdown 预览
- 纯文本编辑体验优化
- 失败态文案统一

验收标准：
- 首页从“能用”提升到“顺手”
- 常用操作反馈更清晰

阶段 G：测试与回归体系补齐

目标：
- 建立“结构可扩展后，对应也要可回归”的检查基础

建议覆盖：
- 页面是否正常渲染
- 后台分享管理是否正常出列表
- 删除分享/清理过期分享是否触发正确状态变化
- 首页基本交互是否正常
- 分享页在空 token / 有 token / 密码保护下的表现

建议形式：
- 继续沿用已有 `node --check` 与 `lint`
- 补前端页面级回归脚本
- 补一批最小可维护的 UI 检查脚本

验收标准：
- 关键页面至少有最小回归覆盖
- 每次加功能后，不必完全手工回看全部页面

阶段 H：统一收口优化

目标：
- 在主体功能回补后，做一轮整体收口

重点内容：
- 清理历史乱码文案
- 统一按钮文案风格
- 统一变量命名
- 清理重复 helper
- 清理旧逻辑残留
- 优化移动端和窄屏布局
- 优化后台页面信息密度与视觉层级

验收标准：
- 代码结构更整洁
- 文案更一致
- UI 细节更像成品

五、当前优先级建议

如果目标是“阶段 E 已完成后，继续推进产品能力”，建议优先级如下：

1. 先做阶段 F：首页核心功能增强
- 后台阶段已经形成可用雏形
- 接下来更适合回到首页用户核心操作体验

2. 同步为阶段 G 做最小回归准备
- 至少把 mock 预览模式和关键页面渲染纳入后续检查范围

3. 后面再做阶段 H 的统一收口
- 等主体功能补到一定程度后再统一收口更划算

六、当前不建议做的事情

1. 不建议现在把所有功能一口气全补回。

2. 不建议为了加快开发，把新逻辑重新塞回 `public/index.js`。

3. 不建议现在就进行大规模命名重构。

4. 不建议在没有 mock 或预览数据支持下直接大修 UI。
