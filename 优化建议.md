## 第二批优化建议（第二轮审查发现）

以下为第二轮深度审查发现的优化点，尚未实施：

### 🔴 P1 — 高影响 / 中低难度

#### 8. D1 限流 → 内存计数器

- **文件**: `functions/api/lib/rate-limiter.js`
- **问题**: 每请求写 D1 `INSERT ... ON CONFLICT ... RETURNING`，D1 按查询计费且有 5-30ms 延迟，高流量时成本高
- **建议**: Worker 模块级 `Map` + TTL 实现内存限流，零成本、亚毫秒延迟；唯一缺点是冷启动丢失状态，可容忍（需 120 次/分窗口内冷启动才可能绕过）

#### 9. 缓存 HMAC CryptoKey

- **文件**: `functions/api/lib/secrets.js:35-41`
- **问题**: `crypto.subtle.importKey` 在每个认证请求的 `signHmac` / `verifyHmac` 中都调用，生成 CryptoKey 是异步且较重的操作
- **建议**: 将生成的 CryptoKey 缓存到 `env._hmacSignKey` / `env._hmacVerifyKey`，避免重复 import

#### 10. ZIP 路径并行解析

- **文件**: `functions/api/lib/zip-download.js:129-138`
- **问题**: `for (const rawPath of rawPaths)` 串行解析，每个路径需 D1 查询 + storage API 调用
- **建议**: 改为 `Promise.all(rawPaths.map(...))` 并行解析

#### 11. 概率性日志清理

- **文件**: `functions/api/lib/common.js:180,193`
- **问题**: `addLog` 每次写入后都调用 `cleanupLogs`（2-3 次 D1 查询），每次文件上传/删除/重命名等操作都额外开销
- **建议**: 改成 ~5% 概率执行清理，或在专门的定时任务中处理

---

### 🟡 P2 — 中影响 / 低难度

#### 12. 定时安全密码比较

- **文件**: `functions/api/lib/auth.js:181`
- **问题**: `username === env.ADMIN_USERNAME && password === env.ADMIN_PASSWORD` 使用 `===` 字符串比较，短路上第一个字符即返回，存在定时侧信道泄露
- **建议**: 使用 `crypto.subtle.timingSafeEqual` 对用户名和密码分别做定时安全比较

#### 13. `api_rate_limits` 加 `window_start` 索引

- **文件**: `migrations/`（需新增 `0003_add_rate_limit_index.sql`）
- **问题**: 1% 概率的清理查询 `DELETE FROM api_rate_limits WHERE window_start < ?` **全表扫描**，高流量下每次清理都很慢
- **建议**: 新增迁移 `CREATE INDEX IF NOT EXISTS idx_rate_limits_window ON api_rate_limits(window_start)`

#### 14. `loadWebhookEndpoints` 加缓存

- **文件**: `functions/api/lib/router.js:70,83,96`，`webhooks.js`
- **问题**: 每次文件操作（上传、删除、重命名等）都调用 `loadWebhookEndpoints(env)` 查询 D1 的 `kv_config`，即使未配 webhook 也有一次 D1 读
- **建议**: 缓存结果到 `env._webhookEndpoints` 带 TTL，或先检查一个布尔标志位

#### 15. `parseCookie` 去重

- **文件**: `auth.js:35-42`、`shares.js:80-87`、`protected-paths.js:91-99`
- **问题**: 三段完全相同的 `parseCookie` 函数重复实现
- **建议**: 抽取到 `common.js` 统一引用

#### 16. `ensureTaskTable` 加缓存

- **文件**: `functions/api/lib/tasks.js:16,213,260,317`
- **问题**: `ensureTaskTable` 每次调用都执行 `CREATE TABLE IF NOT EXISTS`，无任何缓存机制（而 `schema.js` 的 `ensureCoreTables` 使用了 WeakSet）
- **建议**: 使用模块级布尔值或 WeakSet 缓存表就绪状态

#### 17. `storageUsage` 并行迭代

- **文件**: `functions/api/lib/storage.js:231-238`
- **问题**: `for (const space of config.spaces)` 串行调用 `getIndexedStorageUsed`，N 个空间串行 N 次 D1 查询
- **建议**: 改为 `Promise.all(config.spaces.map(...))` 并行

#### 18. 登录失败查询合并

- **文件**: `functions/api/lib/auth.js:99-114`
- **问题**: `recordLoginFailure` 先 `INSERT ... ON CONFLICT` 再单独 `SELECT attempts`，两次 D1 查询
- **建议**: 用 `RETURNING attempts` 合并为一次查询（限流器已使用相同模式）

### 🟢 P3 — 低影响 / 重构型

#### 20. 缩略图避免递归 API 调用

- **文件**: `functions/api/lib/thumbnails.js:79-81`
- **问题**: 缩略图构造 `/api/preview/...` URL 再 fetch，经过完整的鉴权/限流/路由管线；其实可直调 `storageGet`
- **建议**: 跳过路由管线，直接调用底层 storage 读取

#### 21. Route Map 替代 if-else 链

- **文件**: `functions/api/lib/router.js:114-305`
- **问题**: `resolveAdminRoute` 包含 ~30 个 `if/else if` 顺序匹配，每次请求 O(n)
- **建议**: 用 `Map<string, Handler>` 按 `method:path` 做 O(1) 查找

#### 22. Webhook SSRF 防护

- **文件**: `functions/api/lib/webhooks.js:242-253`
- **问题**: `fetch(url)` 的 URL 来自 Admin 配置，未校验内网地址（`127.0.0.1`, `169.254.169.254`, `10.x.x.x` 等），且默认跟随重定向
- **建议**: 添加 `new URL(url).hostname` 私有地址校验

#### 23. 过期分享 / 前缀删除 N+1 查询

- **文件**: `shares.js:238-250`、`file-index.js:276-292`
- **问题**: 过期分享逐行 `DELETE`；`deleteFileIndexPrefix` 逐行 `removeStorageUsage`（每行含 SELECT + 条件 DELETE）
- **建议**: 使用 `D1.batch()` 或单条 `WHERE token IN (...)` 批量删除

#### 24. 缩略图 Content-Type 校验

- **文件**: `functions/api/lib/thumbnails.js:84-91`
- **问题**: 原始图片获取失败时回退到 `originalImageResponse`，但未校验返回的 Content-Type 是否为图片类型
- **建议**: 缓存前检查 Response Content-Type 是否以 `image/` 开头
