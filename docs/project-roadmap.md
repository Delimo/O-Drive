# O-Drive 后续功能与代码优化建议

> 创建时间：2026-07-02。本文基于当前 O-Drive 代码库状态整理，面向后续排期、重构和产品演进。
> 项目当前形态是 Cloudflare Pages Functions + R2 + D1 + 原生 ES Module 前端，不是 React / CRA / Ant Design 项目。
> 本文不是必须一次性完成的 TODO，而是按价值和风险排序的路线图。每项落地后，应把仍有维护价值的结论合并进 `maintenance-handoff.md` 或对应主题文档。

## 总览

| 编号 | 建议 | 优先级 | 主要价值 | 涉及区域 |
| --- | --- | --- | --- | --- |
| 1 | 大目录操作任务化 | 高 | 降低超时和半完成风险 | API、任务系统、前端操作反馈 |
| 2 | 索引一致性检查与修复 | 高 | 提高 R2/D1 长期可靠性 | R2、D1、维护后台 |
| 3 | D1 限流原子化与计数器封装 | 中高 | 提升高并发安全性 | rate-limiter、D1 helper |
| 4 | WebDAV 测试补齐 | 中高 | 降低第三方客户端兼容风险 | WebDAV、R2、回收站 |
| 5 | 后台可观测性增强 | 中 | 让安全、任务、Webhook 状态可见 | 管理后台、通知、统计 |
| 6 | 分享功能产品化 | 中 | 提升实用性和管理能力 | 分享、访问日志、限额 |
| 7 | 测试 mock 降低脆弱性 | 中 | 降低测试维护成本 | test helper、D1 mock |

推荐排期顺序：

1. 先做 **大目录操作任务化** 和 **索引一致性检查/修复**。这两项最影响数据可靠性。
2. 再做 **D1 限流原子化** 和 **WebDAV 测试补齐**。这两项偏安全和边界稳定。
3. 然后做 **后台可观测性** 和 **分享产品化**。这两项更偏体验和运营能力。
4. 穿插做 **测试 mock 优化**。每次新增 D1/任务/维护能力时顺手收敛测试基础设施。

## 1. 大目录操作任务化

> 当前状态（2026-07-04）：✅ 已完成当前验收范围。后端 `file_tasks` 已覆盖 `paste`、`delete`、`zip_download`，前端大批量删除/大粘贴会在预估为大操作时创建后台任务，并刷新任务与通知状态。清空回收站、大目录恢复、任务断点恢复仍保留为后续增强。

### 背景

当前项目已经有后台任务系统，例如 ZIP 下载可以进入任务表，后台系统 Tab 也能展示和重试任务。与此同时，部分重型文件操作仍然走同步 API 链路，例如目录复制、目录移动、批量删除、清空回收站等。

这些操作在小目录上体验很好，但在大目录、深层目录或 R2/D1 写入较慢时容易遇到几个问题：

- Pages Functions 单次请求时间有限，长操作可能超时。
- 操作中途失败时容易产生部分完成状态，虽然当前 `copyTree` 已经能返回子项失败，但用户仍然需要理解和处理结果。
- 前端需要等待请求完成，期间进度感弱。
- 同一类长操作散落在不同 handler 里，不利于统一暂停、重试、审计和恢复。

### 建议方案

把以下操作逐步统一为后台任务：

- 目录复制和目录移动。
- 大批量删除。
- 清空回收站。
- 大目录回收站恢复。
- 大目录 ZIP 或分享打包。
- 后续可能增加的索引重建、孤儿对象清理等维护操作。

实现上可以复用现有 `file_tasks` 思路：

- 小操作仍然同步完成，保持轻量体验。
- 当预估对象数超过阈值，或请求明确要求后台执行时，创建任务并返回 task id。
- 任务记录保存：
  - 操作类型，例如 `paste`、`delete`、`trash_clear`、`restore`。
  - 输入参数，例如 paths、targetDir、strategy。
  - 进度数据，例如 total、completed、failed。
  - 部分失败列表，例如 path、message、阶段。
  - 可恢复状态，例如 cursor、current batch、last processed key。
- 前端通过现有任务列表或轮询接口展示进度。

### 前端体验建议

- 对小操作维持当前即时反馈。
- 对大操作弹出确认框，提示“将转入后台任务”。
- 成功创建任务后，不阻塞当前文件列表，可在通知中心或系统 Tab 查看进度。
- 任务完成后，提供“查看结果”和“定位失败项”。
- 对部分失败任务，提供复制失败列表、重试失败项两个入口。

### 验收标准

- 大目录复制/移动不会因为单次请求时间过长而失败。
- 任务进度能展示 total、completed、failed。
- 失败项能保留具体路径和原因。
- 重试不会重复处理已经成功的项。
- `npm test` 覆盖同步路径、任务化路径、部分失败、重试。

### 风险点

- 任务幂等性要仔细设计，尤其是 move 操作。
- 批量删除和回收站恢复涉及源/目标状态变化，重试逻辑不能简单重复执行。
- 前端任务状态刷新不能造成过多 API 请求。

## 2. 索引一致性检查与修复

### 背景

O-Drive 现在越来越依赖 D1 索引：

- `file_index` 记录用户路径、对象引用、大小和元信息。
- `storage_objects` 支撑内容去重和引用计数。
- `trash` / `trash_entries` 记录回收站状态。
- R2 保存真实对象。

这种结构很适合提升查询效率和实现去重，但长期运行后，任何中途失败、手动改 R2、迁移中断、旧版本遗留数据，都可能造成 R2 和 D1 不一致。

### 建议检查项

建议在后台维护里增加“索引健康检查”，至少检查：

- `file_index` 指向的 `object_key` 在 R2 中不存在。
- R2 中存在对象，但没有任何 `file_index`、`trash` 或系统任务引用。
- `storage_objects.ref_count` 与 `file_index` 实际引用数不一致。
- `file_index.storage_id`、`object_key` 为空或异常。
- `trash` 记录存在，但 trash object 或原对象引用缺失。
- `.system/` 下任务产物过期但未清理。
- 隐藏系统对象被用户路径误引用。

### 修复策略

建议分成两级：

1. **只读检查**
   - 默认只生成报告，不修改数据。
   - 报告包括类别、数量、样例路径、建议动作。

2. **显式修复**
   - 对明确安全的情况自动修复，例如重算 `storage_objects.ref_count`。
   - 对可能删除数据的情况必须二次确认，例如清理 R2 孤儿对象。
   - 所有修复动作写入日志和系统通知。

### API 和后台建议

可以新增维护动作：

- `scan-index-consistency`
- `repair-storage-ref-counts`
- `cleanup-orphan-r2-objects`
- `rebuild-file-index-from-r2`

后台展示：

- 健康分数或状态：正常、警告、需要处理。
- 最近一次扫描时间。
- 各类别问题数量。
- 样例列表和修复按钮。

### 验收标准

- 能检测出断链索引、孤儿 R2 对象、引用计数不一致。
- 默认扫描不改数据。
- 修复动作有日志、通知和测试覆盖。
- 大桶扫描支持分页和任务化，不能一次性列完整桶导致超时。

### 风险点

- “孤儿对象”判断必须保守，避免误删系统任务结果、缩略图缓存、旧迁移对象。
- 从 R2 反向重建索引时可能缺少原始上传元信息，需要明确降级字段。

## 3. D1 限流原子化与计数器封装

### 背景

当前 API / WebDAV 全局限流已经接入 D1 `api_rate_limits` 表，相比内存 Map 更可靠。但当前实现仍是常见的读写流程：

1. 读取当前窗口和计数。
2. 判断是否跨窗口或超限。
3. 插入或更新计数。

在极高并发下，这种 `SELECT -> UPDATE` 仍可能出现计数竞争。对一般自托管云盘来说问题不大，但如果把它作为安全边界，后续可以继续收紧。

### 建议方案

抽出一个统一的 D1 计数器 helper，例如：

- `incrementFixedWindowCounter(env, table, key, windowMs, limit)`
- `checkFixedWindowLimit(env, key, options)`
- `cleanupExpiredCounters(env, table, windowMs)`

再把以下能力统一到这个 helper：

- API 全局限流。
- WebDAV 限流。
- 登录失败账号维度软降速。
- 下载突增检测中可复用的窗口计数部分。
- 后续分享链接访问限额。

### 原子化方向

优先考虑 D1 支持良好的 SQL 形态：

- 使用 `INSERT ... ON CONFLICT DO UPDATE` 合并插入和递增。
- 在同一条 SQL 中处理跨窗口重置。
- 返回更新后的计数和窗口起点。

如果 D1 的返回能力不足，可以选择“足够紧凑”的两步实现，但封装统一入口，便于以后替换。

### 验收标准

- API 和 WebDAV 限流行为不变。
- 高并发单元测试或模拟测试中，计数不会明显低估。
- helper 有独立测试。
- 所有限流响应保留 `Retry-After`。

### 风险点

- D1 SQL 方言和 SQLite 有差异，需要以 Wrangler / D1 实际行为为准。
- 过度追求原子化可能让 SQL 难读，建议先封装清楚，再逐步优化。

## 4. WebDAV 测试补齐

### 背景

WebDAV 是第三方客户端入口，客户端实现差异很大。Windows、macOS、iOS、Cyberduck、rclone 等客户端对路径编码、覆盖语义、目录操作、错误码的处理都不完全一致。

当前 WebDAV 基础测试已经覆盖认证、GET、PUT、MKCOL、MOVE、COPY、限流等，但仍有一些风险面可以补齐。

### 建议补充场景

优先补这些测试：

- `DELETE` 文件进入回收站，目前相关测试仍有 skip。
- 删除目录时，目录内文件都进入回收站，且可恢复。
- `MOVE` 目录到已有目标时，`Overwrite: F` 返回 412。
- `MOVE` 目录到已有目标时，`Overwrite: T` 正确覆盖或先软删除目标。
- `COPY` 目录复制索引和 R2 引用正确。
- `Destination` 是完整 URL、相对路径、带中文/空格/百分号编码时都能解析。
- 保留路径如 `.system/`、`.thumbs/` 被拒绝。
- Basic Auth 失败不会执行写操作。
- 限流发生在认证前，错误码和 `Retry-After` 正确。

### 兼容性建议

- WebDAV 错误响应尽量使用常见客户端能理解的状态码。
- 对路径解码和规范化集中封装，避免不同 method 各自处理。
- 对 `Depth`、`Overwrite`、`Destination` 等头部做更明确的解析和测试。

### 验收标准

- `tests/webdav.test.mjs` 覆盖 DELETE、目录 MOVE/COPY、覆盖、编码路径。
- 取消现有可修复的 skip。
- WebDAV 写操作和普通 API 写操作在索引、回收站、日志方面保持一致。

### 风险点

- 不同客户端可能依赖非标准行为，改状态码或响应体前最好保守。
- WebDAV 目录语义和 O-Drive 的逻辑目录/`.folder` 占位文件之间要保持一致。

## 5. 后台可观测性增强

### 背景

当前系统已经有日志、通知、Webhook 投递记录、任务状态、系统警告、限流和登录失败数据。问题是这些信息分散在不同表和不同后台区域中，管理员需要自己拼信息。

可以新增一个“安全与系统事件”视角，让后台更像一个可运维工具。

### 建议展示内容

可以在系统 Tab 或概览 Tab 中加入：

- 最近 24 小时 API 限流触发次数。
- 最近 24 小时 WebDAV 限流触发次数。
- 登录失败最多的 IP。
- 登录失败最多的账号维度。
- 当前处于锁定/降速风险的登录来源。
- Webhook 最近失败率和连续失败 endpoint。
- 后台任务失败率、部分失败任务数。
- 索引一致性最近扫描结果。
- 存储容量告警、任务失败告警、系统警告统一列表。

### 数据来源建议

短期可以复用现有表：

- `logs`
- `system_warnings`
- `notifications`
- `webhook_deliveries`
- `file_tasks`
- `login_attempts`
- `api_rate_limits`

后续如果要保留长期趋势，可以新增轻量聚合表，例如 `system_metrics_daily`，避免每次后台打开都扫大表。

### 前端建议

- 不要做花哨大屏，保持后台工具风格。
- 用紧凑的 stat card、表格和筛选器。
- 对异常项提供直接跳转，例如 Webhook endpoint 失败跳到 Webhook 投递列表。
- 对维护建议提供明确按钮，例如“运行索引检查”。

### 验收标准

- 管理员打开后台即可看到系统是否健康。
- 异常项能定位到具体来源。
- 不引入明显慢查询。
- 有核心渲染测试和 API 测试。

### 风险点

- 统计查询容易越做越重，必须加时间窗口和 limit。
- 不要把敏感信息直接展示给非管理员角色。

## 6. 分享功能产品化

> 当前状态（2026-07-04）：✅ 已完成当前验收范围。已增加访问次数、最近访问、访问日志、后台分享列表展示和 90 天日志保留；下载次数仍使用原子计数避免并发超限。`max_bytes`、二维码、公开上传收件箱、备注/标签和访问通知仍保留为后续产品增强。

### 背景

当前分享已经支持文件、文件夹、bundle、密码保护、过期保留和重新启用。这已经是很实用的基础能力。下一步可以围绕“可控、可审计、可运营”继续增强。

### 功能建议

优先级较高的方向：

- 每个分享链接的访问次数、下载次数、最近访问时间。
- 分享访问日志：时间、IP、User-Agent、访问路径、是否成功。
- 单分享下载次数上限。
- 单分享总流量或总下载字节上限。
- 单分享限速或突增下载保护。
- 批量禁用、批量删除过期分享。
- 分享备注或标签，方便后台管理。

中长期方向：

- 公开上传收件箱：外部用户通过分享链接上传文件到指定目录。
- 分享页品牌/说明文本配置。
- 分享链接二维码。
- 分享链接访问通知。

### 数据模型建议

可以扩展或新增：

- `share_links.visit_count`
- `share_links.download_count`
- `share_links.last_accessed_at`
- `share_links.max_downloads`
- `share_links.max_bytes`
- `share_access_logs`

访问日志建议做保留期，避免无限增长。

### 前端建议

后台分享列表展示：

- 访问次数。
- 下载次数。
- 最近访问。
- 剩余下载次数。
- 是否达到上限。

分享详情弹窗展示：

- 访问趋势摘要。
- 最近访问记录。
- 快速禁用按钮。

### 验收标准

- 分享访问和下载能准确记录。
- 达到上限后返回明确错误。
- 后台能看到访问/下载统计。
- 访问日志有保留期或清理机制。

### 风险点

- 访问日志可能包含 IP 和 User-Agent，需注意隐私和清理策略。
- 下载计数与 range 请求、预览请求要区分清楚，避免一次下载被重复计数。

## 7. 测试 mock 降低脆弱性

### 背景

`tests/helpers/make-env.mjs` 当前承担了大量 Cloudflare 环境模拟，包括 R2、D1、任务、通知、分享、索引、Webhook 等。它的优点是速度快、覆盖广；缺点是 D1 行为大量依赖 SQL 正则匹配，功能越多越容易变脆。

后续继续扩展 D1 表、维护任务和统计查询时，测试 helper 会越来越难维护。

### 建议方向

分阶段降低脆弱性：

1. **拆分 make-env**
   - R2 mock。
   - D1 statement mock。
   - 文件索引 mock。
   - 分享/任务/通知 fixture helper。

2. **抽出 SQL handler registry**
   - 当前很多 `if (/SQL/i.test(sql))` 可以集中成注册表。
   - 每个 handler 写清匹配 SQL、读写表、返回形态。

3. **增加接近真实 SQLite 的测试层**
   - 对 schema、迁移、复杂查询，用真实 SQLite 或更接近 D1 的本地执行层。
   - 普通模块测试仍保留快速 mock。

4. **统一 fixture 工厂**
   - 例如 `makeIndexedFile()`、`makeShare()`、`makeTrashEntry()`。
   - 减少测试里手写重复对象。

### 验收标准

- 新增 D1 查询时，不需要在一个超长文件里到处插入正则。
- 测试失败能更快定位是哪张模拟表或哪条 SQL。
- 保持现有测试速度，不让所有测试都变成慢集成测试。

### 风险点

- 一次性重写 test helper 风险很高，建议随着新功能逐步拆。
- 真实 SQLite 和 Cloudflare D1 仍可能有差异，不能完全替代 D1 行为验证。

## 综合建议

如果只做三件，建议优先：

1. **大目录操作任务化**：直接提升大文件/大目录场景的可靠性。
2. **索引一致性检查与修复**：为长期运行兜底，避免 R2/D1 漂移越来越难处理。
3. **后台可观测性增强**：让管理员能看见系统问题，而不是等用户报告。

如果只做一件，优先做 **索引一致性检查与修复**。因为 O-Drive 已经进入“D1 索引驱动 R2 数据”的阶段，索引健康会成为长期稳定性的底盘。
