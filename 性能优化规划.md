# O-Drive 性能优化规划

> 基于前后端全量代码分析，按优先级整理。

---

## P0 — 阻塞级（严重影响核心体验）

### 1. 文件列表分页（后端 + 前端）

**问题**
- 后端 `functions/api/lib/file-reads.js:221` — `storageList` 不带 `limit`/`cursor`，一次性返回目录下所有文件，大目录（500+ 文件）全量加载
- 前端 `public/js/render/home.js:241,322` — 所有条目全部渲染到 DOM，无虚拟滚动

**方案**
- 后端：`handleListFiles` 增加 `?limit=200&cursor=xxx` 参数，返回 `nextCursor`
- 前端：改为按需加载 + 滚动到底部自动加载下一页
- 保留排序和搜索在当前页内进行

**涉及文件**
- `functions/api/lib/file-reads.js`
- `public/js/render/home.js`
- `public/js/render/shared.js`
- `public/js/state/` 相关 slice

### 2. 全页重渲染改为局部渲染

**问题**
- `public/index.js:256-290` — 任何状态变化（搜索输入、筛选、loading）都重新生成整个页面 HTML 模板，再 diff 替换
- 每个 thunk 的 `loading=true/false` 触发两次全页重渲染

**方案**
- 将页面拆成独立渲染区域：面包屑、文件列表、详情面板、模态框、toast
- 每个区域只订阅自己依赖的状态切片
- loading 状态只影响对应区域

**涉及文件**
- `public/index.js`
- `public/js/render/home.js`
- `public/js/render/shared.js`
- `public/js/state/store.js`

## P1 — 严重影响（日常操作频繁触发）

### 3. 隐藏/受保护路径加进程级缓存

**问题**
- `functions/api/[[path]].js:92` — 每次 API 请求都查询 D1 获取隐藏路径和受保护路径
- 即使非文件操作（`/api/admin/health`、`/api/auth/role`）也跑两次 D1 查询

**方案**
- 类似 `loadStorageConfig`，把结果缓存在 `env` 对象上
- 管理端新增/删除/修改时清除缓存

**涉及文件**
- `functions/api/[[path]].js`
- `functions/api/lib/request-context.js`
- `functions/api/lib/protected-paths.js`

### 4. 管理后台按 tab 懒加载

**问题**
- `public/index.js:448-466` — 进入后台一次性发 13 个 API 请求拉取所有 tab 数据

**方案**
- 只加载当前 tab 的数据，切换 tab 时再按需加载
- 切回已加载过的 tab 时复用上次数据（或加缓存时间）

**涉及文件**
- `public/index.js`
- `public/js/state/thunks/admin.js`
- `public/js/render/` 管理后台相关

### 5. CSS 合并与去重

**问题**
- `public/index.html:12-23` — 加载 11 个 CSS 文件，阻塞渲染
- `components.css`（47.6KB）与 `components/*.css` 内容重复

**方案**
- 合并为 2-3 个 CSS 文件（核心样式 + 页面专属样式）
- 删除重复的定义
- 考虑用 Tailwind 的 `@layer` 组织

**涉及文件**
- `public/index.html`、`public/admin.html`、`public/share.html`
- `public/css/` 全部

### 6. NOT IN 子查询优化

**问题**
- `common.js:202` — `DELETE FROM logs WHERE id NOT IN (SELECT id FROM ... LIMIT ?)`
- 同样模式：`tasks.js:57`、`webhooks.js:321`
- SQLite 处理 `NOT IN (subquery)` 效率极低

**方案**
- 改为：先用子查询拿到要保留的 id，再 `DELETE WHERE id NOT IN (显式列表)`
- 或改用 `LEFT JOIN ... IS NULL`

**涉及文件**
- `functions/api/lib/common.js`
- `functions/api/lib/tasks.js`
- `functions/api/lib/webhooks.js`

### 7. 通知加索引 / 轮询优化

**问题**
- `notifications` 表完全没有索引（`created_at`、`read`）
- 通知轮询每 30 秒请求一次，即使页面在后台

**方案**
- D1 迁移加索引：`CREATE INDEX idx_notifications_created_at ON notifications(created_at)`
- 轮询时检测 `document.hidden`，后台时暂停
- 未登录时不轮询

**涉及文件**
- `migrations/0001_initial.sql`（或新迁移）
- `functions/api/lib/schema.js`
- `public/index.js`

## P2 — 中等影响（特定场景触发）

### 8. R2 操作优化

**问题**
- `r2-tree.js:84-162` — 复制/移动文件夹时，每个文件先下载再重新上传
- `trash.js:142-156` — 删除到回收站、从回收站恢复，同样下载+重新上传
- 对于大文件或大目录，耗时长且浪费带宽

**方案**
- 小文件（<10MB）继续走内存拷贝
- 大文件改为异步任务（`file_tasks` 表），后台处理
- 或限制单次复制/移动的文件数量

**涉及文件**
- `functions/api/lib/r2-tree.js`
- `functions/api/lib/trash.js`

### 9. 筛选器加防抖

**问题**
- `public/js/events/ui-actions.js:59-87` — 类型/大小/日期筛选器每次变更立即发 API

**方案**
- 与搜索同样的 260ms 防抖

**涉及文件**
- `public/js/events/ui-actions.js`

### 10. 回收站批量操作改并行

**问题**
- `public/js/state/thunks/maintenance.js:87-155` — 批量恢复/删除逐个发请求（N+1）

**方案**
- 后端加批量 API（接受 id 数组）
- 或前端 `Promise.allSettled` 并发请求

**涉及文件**
- `functions/api/lib/trash.js`
- `public/js/state/thunks/maintenance.js`

### 11. `chooseUploadStorage` 减少查询

**问题**
- `functions/api/lib/storage.js:330` — 每次上传都查 `getIndexedStorageUsed`，跑 SUM 查询

**方案**
- 上传路径绑定时，只在接近配额时才精确查询
- 或缓存存储用量（允许短时间滞后）

**涉及文件**
- `functions/api/lib/storage.js`

### 12. 管理员统计页面优化

**问题**
- `admin-stats.js:58` — 当索引不存在时，一次性加载 20000 个 R2 对象到内存

**方案**
- 优先走文件索引；索引不存在时提示重建，不做全量 R2 扫描
- 或分批处理

**涉及文件**
- `functions/api/lib/admin-stats.js`

## P3 — 低优先级（锦上添花）

### 13. 速率限制减少 D1 查询

**问题**
- `rate-limiter.js:44-59` — 每次请求先 `INSERT ... ON CONFLICT` 再 `SELECT`，共 2 次 D1 查询

**方案**
- 用 `INSERT ... RETURNING *` 合并为一次查询（D1 支持）

### 14. 搜索加全文索引

**问题**
- `file-index.js:564` — `LIKE %q%` 无法走索引，全表扫描

**方案**
- D1 不支持 FTS5，可考虑外部搜索服务
- 或限制搜索范围（限定目录、类型等）
- 或缓存热门搜索结果

### 15. 局部渲染替代 morphdom

**问题**
- 当前用 HTML 字符串拼接 + morphdom diff

**方案**
- 改为直接操作 DOM（`innerHTML` 替换局部容器）
- 或引入轻量框架的响应式方案

### 16. `system_warnings` 加索引

**涉及文件**
- `migrations/0001_initial.sql`
- `functions/api/lib/schema.js`

### 17. `ensureCoreTables` 并发保护

**涉及文件**
- `functions/api/[[path]].js`

## 执行优先级

```
第一阶段（体感提升最明显）
  ├── P0-1 文件列表分页
  ├── P0-2 局部渲染
  └── P1-4 管理后台懒加载

第二阶段（日常操作更流畅）
  ├── P1-3 隐藏/受保护路径缓存
  ├── P1-5 CSS 合并去重
  ├── P2-9 筛选器防抖
  └── P2-10 批量操作改并行

第三阶段（后台/数据库优化）
  ├── P1-6 NOT IN 子查询优化
  ├── P1-7 通知加索引 + 轮询优化
  ├── P2-8 R2 操作优化
  └── P2-11 upload 存储用量查询优化

第四阶段（进一步打磨）
  └── P3 各项低优先级优化
```
